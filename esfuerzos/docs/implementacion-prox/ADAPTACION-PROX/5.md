# Paso 5 — Intake: parseo, hashing y commit de Report + Photos

Crea los registros `Report` y `Photo` en DB al completar el flujo familiar. El teléfono nunca se almacena en texto plano.

---

## Archivos modificados

**`app/core/intake.py`** — nuevo módulo con tres funciones:

| Función | Descripción |
|---|---|
| `hash_phone(phone)` | SHA-256 del teléfono → `reporter_wa_hash` |
| `parse_person_data(raw)` | Parte el mensaje por comas: `full_name / gender / age / last_seen_location`. Tolerante a partes faltantes. |
| `commit_report(db, conversation, client_phone, notes)` | Lee `context["intake_person_raw"]` y `context["pending_photos"]`, crea `Report` + `Photo`s, limpia ambas claves y hace commit. |

**`app/models/reporte.py`** — dos campos nuevos:
- `Report.gender` (`String(20)`, nullable) — masculino | femenino | otro
- `Photo.local_path` (`Text`, nullable) — ruta local tras descarga

**`app/bot/orchestrator.py`** — dos hooks entre Paso 9 y Paso 10:

```
guia_familiar → pedir_foto   : context["intake_person_raw"] = message_text
→ reporte_guardado           : commit_report(notes=message_text)
```

## Pendientes

### Flujo `found` (rescatista / hospital)

`commit_report` siempre crea `kind="missing"`. Cuando se implementen los flujos de rescatista y hospital, el `kind` debe derivarse del perfil detectado en `bienvenida` (guardar `context["intake_kind"]` al seleccionar opción 2 o 3).

### Parseo robusto

`parse_person_data` confía en que el usuario sigue el formato CSV. Para producción evaluar un paso LLM ligero o expresiones regulares más sofisticadas que toleren variantes como "34 años" o "Cumaná, Sucre".

### Migración de columnas

`gender` (Report) y `local_path` (Photo) son columnas nuevas. En bases de datos existentes ejecutar:
```sql
ALTER TABLE reports ADD COLUMN gender VARCHAR(20);
ALTER TABLE photos ADD COLUMN local_path TEXT;
```
En dev con SQLite basta con eliminar el archivo `.db` y dejar que `Base.metadata.create_all()` lo recree.
