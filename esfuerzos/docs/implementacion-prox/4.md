# Paso 4 — Sesión multi-foto con TTL

Reemplaza el mecanismo "listo" por avance automático basado en TTL y conteo máximo. El usuario solo envía fotos; el bot avanza solo.

---

## Archivos modificados

**`app/config.py`** — Tres nuevas variables:

| Variable | Default | Descripción |
|---|---|---|
| `photo_max_count` | `5` | Fotos máximas por sesión; al alcanzarlo avanza inmediatamente |
| `photo_ttl_seconds` | `60` | Segundos sin nueva foto para cerrar la sesión al próximo texto |
| `photo_storage_path` | `media/photos` | Directorio local de descarga |

**`app/bot/flow_seeder.py`** — `pedir_foto.next_node_map` reducido a `{"default": "pedir_foto"}`. "listo" eliminado.

**`app/bot/orchestrator.py`** — Tres métodos nuevos:

- `_handle_pedir_foto`: intercepta toda interacción en nodo `pedir_foto`. Si llega `media_url` descarga la foto y acumula en `context["pending_photos"]`; si llega texto evalúa TTL.
- `_advance_from_foto`: navega a `notas_adicionales`, genera respuesta del template.
- `_download_photo`: GET httpx al `media_url`, detecta ext por Content-Type, guarda en `{photo_storage_path}/{conversation_id}_{index}.{ext}`.

---

## Lógica de avance

```
Foto recibida
  → descargar + guardar en pending_photos
  → count >= photo_max_count → avanzar (motivo: max_alcanzado)
  → si no → responder "N/max recibidas"

Texto recibido (sin fotos aún)
  → responder "aún no has enviado fotos"

Texto recibido (con fotos)
  → elapsed < photo_ttl_seconds → responder "aún puedes enviar más"
  → elapsed >= photo_ttl_seconds → avanzar (motivo: ttl_expirado)
```

Las fotos se almacenan en `context["pending_photos"]` como lista de objetos `{media_url, local_path, received_at}`. El `report_id` se asigna en el Paso 5.

## Cómo probar el Paso 4

Mismo setup del Paso 1. Arranca desde `guia_familiar` (envía "1" tras el saludo).

```bash
BASE="http://localhost:8000/webhook/waha"
PHONE="584121234567@c.us"

# 1. Hola → bienvenida
curl -s -X POST $BASE -H "Content-Type: application/json" \
  -d "{\"session\":\"default\",\"event\":\"message\",\"payload\":{\"from\":\"$PHONE\",\"fromMe\":false,\"body\":\"hola\",\"hasMedia\":false}}"

# 2. Seleccionar familiar
curl -s -X POST $BASE -H "Content-Type: application/json" \
  -d "{\"session\":\"default\",\"event\":\"message\",\"payload\":{\"from\":\"$PHONE\",\"fromMe\":false,\"body\":\"1\",\"hasMedia\":false}}"

# 3. Datos de la persona
curl -s -X POST $BASE -H "Content-Type: application/json" \
  -d "{\"session\":\"default\",\"event\":\"message\",\"payload\":{\"from\":\"$PHONE\",\"fromMe\":false,\"body\":\"María García, femenino, 34, Cumaná centro\",\"hasMedia\":false}}"
# Esperado: pedir_foto

# 4. Primera foto
curl -s -X POST $BASE -H "Content-Type: application/json" \
  -d "{\"session\":\"default\",\"event\":\"message\",\"payload\":{\"from\":\"$PHONE\",\"fromMe\":false,\"body\":\"\",\"hasMedia\":true,\"mediaUrl\":\"http://waha/media/foto1.jpg\"}}"
# Esperado: "Imagen recibida (1/5). Puedes enviar más fotos."

# 5. Segunda foto
curl -s -X POST $BASE -H "Content-Type: application/json" \
  -d "{\"session\":\"default\",\"event\":\"message\",\"payload\":{\"from\":\"$PHONE\",\"fromMe\":false,\"body\":\"\",\"hasMedia\":true,\"mediaUrl\":\"http://waha/media/foto2.jpg\"}}"
# Esperado: "Imagen recibida (2/5). Puedes enviar más fotos."

# 6. Texto durante TTL activo
curl -s -X POST $BASE -H "Content-Type: application/json" \
  -d "{\"session\":\"default\",\"event\":\"message\",\"payload\":{\"from\":\"$PHONE\",\"fromMe\":false,\"body\":\"ya terminé\",\"hasMedia\":false}}"
# Esperado: "Aún puedes enviar más fotos (2/5 recibidas)."

# 7. Esperar TTL (60s por defecto) y enviar cualquier texto
sleep 65
curl -s -X POST $BASE -H "Content-Type: application/json" \
  -d "{\"session\":\"default\",\"event\":\"message\",\"payload\":{\"from\":\"$PHONE\",\"fromMe\":false,\"body\":\"ok\",\"hasMedia\":false}}"
# Esperado: notas_adicionales
```

## Pendientes

### Fotos sin `report_id`

Las fotos se acumulan en `context["pending_photos"]` porque el `Report` aún no existe. El Paso 5 (`intake.py`) creará el `Report` y vinculará cada foto como registro `Photo` en base de datos.

### Descarga real de WAHA

`_download_photo` hace GET directo al `mediaUrl` del payload. En producción WAHA requiere el header `X-Api-Key` para descargar media. Agregar al cliente httpx cuando se configure WAHA real.

### TTL en producción

El TTL actual se evalúa en el momento del próximo mensaje. Si el usuario nunca vuelve a escribir, el nodo queda en `pedir_foto` indefinidamente. Para producción: tarea periódica (APScheduler o Celery beat) que evalúe conversaciones varadas y las avance.
